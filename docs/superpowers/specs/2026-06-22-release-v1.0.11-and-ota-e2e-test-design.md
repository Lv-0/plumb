# 发布 v1.0.11 + 端到端 OTA 测试

**日期：** 2026-06-22
**类型：** 发布 + 验证（非新功能）
**目标版本：** v1.0.11
**当前线上版本：** v1.0.10

## 目标

1. 把当前未提交的 OTA shell-quoting 修复（`UpdateCoordinator` / `UpdateInstaller` / 测试）+ `release_build.sh` 改进，随 **v1.0.11** 发布上线。
2. 在本机（当前装 v1.0.10）走完一次**真实端到端 OTA**：菜单检查更新 → 下载 → sha256 校验 → 解压 → 旧 app 退出 → 新 app 以 installer 模式启动 → 系统密码框 → `rm -rf && cp -R` 替换 `/Applications/Plumb.app` → 重启。
3. 验证替换后 `/Applications/Plumb.app` 真的变成 1.0.11，且新二进制包含 shell-quoting 修复。

## 非目标

- 不做 Developer ID 签名 / notarization（本机无证书；沿用 v1.0.6–1.0.10 的 self-signed 路径）。
- 不重打 dmg（OTA 只用 zip；dmg 复用现有 1.0.10 版本）。
- 不改 OTA 架构 / 不新增功能代码。

## 执行计划

### 阶段 1：准备（commit + build）

**1.1 跑单元测试**（先验证未提交改动不回归）
```

```
- 期望全绿（含新增的 shell-quoting 测试）。
- 若失败 → 停下，进入 systematic-debugging，不发布。

**1.2 提交改动到 main**（两个 commit）
- Commit A：`fix(ota): shell-quote paths in relaunch & install scripts`
  - 文件：`Sources/Plumb/UpdateCoordinator.swift`, `Sources/Plumb/UpdateInstaller.swift`, `Tests/PlumbTests/UpdateInstallerTests.swift`
- Commit B：`chore(release): sign before dmg + add OTA zip step to release_build.sh`
  - 文件：`scripts/release_build.sh`

**1.3 重打 v1.0.11**（self-signed，方案 A）
```
VERSION=1.0.11 scripts/build_app.sh
VERSION=1.0.11 scripts/create_zip.sh
```
- `build_app.sh` 会用 `Plumb Local Signer`（若存在），否则回退 ad-hoc 签名。两种都 OK。
- 产出 `dist/Plumb.app`（版本 1.0.11）和 `dist/Plumb-1.0.11.zip`。
- 记录新 zip 的 sha256（create_zip.sh 会打印）。

**验证点：**
- `defaults read dist/Plumb.app/Contents/Info.plist CFBundleShortVersionString` → `1.0.11`
- `shasum -a 256 dist/Plumb-1.0.11.zip` 记录下来，与发布后 appcast.json 比对

### 阶段 2：发布到 GitHub

**2.1 调用 publish_release.sh**
```
GITHUB_TOKEN=<用户提供> VERSION=1.0.11 scripts/publish_release.sh v1.0.11
```
脚本执行链：
1. 创建 GitHub Release `v1.0.11`（body 用脚本内硬编码的 v1.0.10 notes —— 见下方**已知瑕疵**）
2. 上传 `dist/Plumb.dmg`（现有 1.0.10 dmg —— 用户已确认复用）
3. 上传 `dist/Plumb-1.0.11.zip`
4. 用新版本号 + 新 sha256 重写 `appcast.json`，commit + push main

**验证点：**
- `curl -sL https://github.com/Lv-0/plumb/releases/download/v1.0.11/Plumb-1.0.11.zip | shasum -a 256` 与 appcast.json 的 sha256 一致
- `git log -1 appcast.json` 显示新 commit
- GitHub Release 页面 v1.0.11 有两个 asset

### 阶段 3：端到端 OTA 验证（用户手动触发）

> 本阶段**由用户在运行中的 v1.0.10 里手动操作**，我只提供步骤和验证命令。

**3.1 重置节流**（可选，若用菜单手动检查可跳过）
```
defaults delete com.comet.plumb otaLastCheckTimestamp
```

**3.2 在 v1.0.10 里触发更新**
- 菜单栏 Plumb 图标 → 检查更新（`AppDelegate.checkForUpdates` → `checkForUpdatesManually`，绕过节流）
- 预期弹窗："有新版本 1.0.11" + notes
- 点"立即更新"

**3.3 预期链路**（自动执行）
1. 下载 `Plumb-1.0.11.zip`
2. sha256 校验
3. 解压到临时位置
4. 写 `installerMode=true` + `installerAppPath`
5. 旧 app 写 detached shell 脚本（`sleep 2; open -n <newApp>`）→ `exit(0)`
6. 新 app（1.0.11）以 installer 模式启动
7. 系统密码框（AppleScript `with administrator privileges`）
8. `rm -rf /Applications/Plumb.app && cp -R <newApp> /Applications/Plumb.app`
9. 清零 `installerMode`，Launch Services 启动新版本

**3.4 验证成功**
```
defaults read /Applications/Plumb.app/Contents/Info.plist CFBundleShortVersionString
# 期望：1.0.11
```
- About 页版本号 → 1.0.11
- 新二进制含 shell-quoting 修复（可 `strings` 或看 git log 确认源码已含）

**3.5 失败排查路径**（若某步失败）
- 无弹窗 / 提示已是最新 → appcast 缓存或节流：删 `otaLastCheckTimestamp` 重试
- 下载失败 → 检查 GitHub release asset 是否公开可下载
- sha256 失败 → appcast.json sha256 与实际 zip 不符
- 密码框后失败 → 看 Console.app 过滤 `Plumb`，或手动跑 `UpdateInstallerCommand.buildShellScript` 的输出验证
- 替换后没重启 → README 已说明：手动 open

## 已知瑕疵（本测试可接受，不阻塞）

1. **Release body 是 v1.0.10 的 notes。** `publish_release.sh` 把 body 硬编码在脚本里（v1.0.10 内容）。v1.0.11 release 页面会显示 v1.0.10 的 changelog。这不影响 OTA（OTA 只读 appcast.json，不读 release body），但用户在 GitHub 看到的 notes 会不准。**接受**，因为本次重点是 OTA 管道测试；若要修，应单独改 `publish_release.sh` 让 body 参数化。
2. **dmg 是 1.0.10 版本。** 用户已确认复用。
3. **Token 暴露。** 见 4.1。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| `swift test` 失败 | 阶段 1.1 停下，先修 |
| build_app.sh 失败（图标/签名） | 看错误；ad-hoc 回退应能继续 |
| GitHub API 限流 / token 权限不足 | publish_release.sh 会报错；检查 token scopes（需 `repo`） |
| OTA 替换 `/Applications/Plumb.app` 需管理员密码 | 用户在 3.2 手动输；代码用 AppleScript 弹系统框 |
| 替换失败导致 `/Applications/Plumb.app` 损坏 | 现有 `dist/Plumb.app`（1.0.11）可手动 cp 回去；或重装 dmg |
